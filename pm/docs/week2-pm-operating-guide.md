# AskLake 2주차 프로젝트 관리자 운영 가이드

작성 기준일: 2026-06-26

## 이 문서의 역할

이 문서는 팀장 겸 프로젝트 관리자가 2주차 AskLake 개발을 어떻게 끌고 가야 하는지 정리한 운영 가이드다.

지금 필요한 역할은 팀원에게 일을 더 많이 시키는 사람이 아니다. 여섯 명이 각자 만든 기능이 하나의 발표 가능한 흐름으로 합쳐지게 만드는 사람이다. 그래서 이 문서는 코드 세부 구현보다 일정, 책임, 위험, 의사결정 기준에 집중한다.

이 문서를 볼 때 가장 먼저 기억해야 할 문장은 하나다.

**이번 주의 목표는 모든 기능을 완성하는 것이 아니라, 사용자가 데이터 연결부터 질문 응답까지 이어지는 하나의 신뢰 가능한 흐름을 보는 것이다.**

## 이번 주에 보여줘야 하는 이야기

AskLake가 보여줘야 하는 핵심 이야기는 단순하다.

사용자가 데이터를 연결한다. 시스템은 데이터를 읽고, 분석 가능한 형태로 정리한다. 실행 결과와 메타데이터가 카탈로그에 남는다. 사용자가 질문하면 시스템은 카탈로그를 근거로 답한다. 화면에서는 실행 상태, 데이터 정보, 질문 결과, 근거를 확인할 수 있다.

이 흐름이 이어지면 발표가 된다. 반대로 데이터셋이 아무리 크고, 도구가 아무리 많아도 이 흐름이 끊기면 발표에서 설명하기 어렵다.

## 데이터셋은 세 개지만, 우선순위는 하나다

현재 데이터셋 전략은 세 갈래다.

첫 번째는 반정형 데이터다. 이 경로는 Amazon Reviews JSON 또는 JSONL을 사용한다. 이 데이터셋은 이번 발표의 핵심 경로다.

두 번째는 정형 데이터다. 이 경로는 TLC NYC Taxi를 사용한다. 이 데이터셋은 정형 대용량 일괄 처리 가능성을 보여주는 보조 증거다.

세 번째는 실시간형 데이터다. 이 경로는 Kaggle의 [Ecommerce Behavior Data from Multi Category Store](https://www.kaggle.com/datasets/mkechinov/ecommerce-behavior-data-from-multi-category-store/)를 Kafka로 재생해서 실시간처럼 들어오는 흐름을 만든다. 이 데이터셋은 실시간형 수집 가능성을 보여주는 보조 증거다.

여기서 중요한 판단은 세 데이터셋을 동시에 같은 깊이로 완성하려고 하면 안 된다는 점이다. 발표에서 반드시 닫아야 하는 경로는 Amazon Reviews JSON이다. Taxi와 Kafka는 핵심 경로가 닫힌 뒤 제품 확장성을 보여주는 증거로 붙여야 한다.

## 핵심 발표 경로

이번 발표의 중심 경로는 아래와 같다.

```text
Amazon Reviews JSON
-> M3 데이터 구조 파악과 변환 정의
-> M5 작업 실행과 카탈로그 저장
-> M6 카탈로그 기반 질문 응답
-> M1 화면 표시
```

이 경로가 AskLake의 본선이다.

M3가 데이터 구조를 파악하지 못하면 M5는 무엇을 실행해야 하는지 흐려진다. M5가 실행 결과와 카탈로그를 안정적으로 남기지 못하면 M6는 근거 있는 답변을 만들 수 없다. M6가 실제 카탈로그를 근거로 답하지 못하면 M1은 데모 화면만 보여주게 된다.

프로젝트 관리자는 매일 이 경로가 어디까지 이어졌는지 확인해야 한다. 다른 작업이 많아도 이 경로가 막혀 있으면 그날의 우선순위는 핵심 경로 복구다.

## Amazon Reviews JSON의 의미

Amazon Reviews JSON은 이번 주 발표에서 가장 중요한 데이터셋이다.

이 데이터셋으로 보여줘야 하는 것은 “리뷰 데이터를 멋지게 분석한다”가 아니다. 더 정확히는, 반정형 데이터가 들어왔을 때 AskLake가 그 구조를 파악하고, 필요한 형태로 정리하고, 실행 결과를 카탈로그에 남기고, 그 카탈로그를 근거로 질문에 답할 수 있음을 보여주는 것이다.

M3는 이 데이터에서 필드, 타입, 누락 가능성, 중첩 구조를 파악해야 한다. 예를 들어 상품별 리뷰 수, 평균 평점, 검증된 구매 여부 같은 값을 만들 수 있는 최소 변환 정의가 필요하다.

M5는 그 변환이 실행되었을 때 실행 상태, 처리 행 수, 출력 위치, 오류 여부 같은 정보를 남겨야 한다. 그리고 최신 성공 결과가 카탈로그에 반영되어야 한다.

M6는 하드코딩된 답변이 아니라 카탈로그에 있는 데이터셋 설명, 스키마, 지표를 보고 어떤 데이터셋을 쓸지 판단해야 한다.

M1은 사용자가 이 흐름을 눈으로 따라갈 수 있게 보여줘야 한다.

## TLC NYC Taxi의 의미

TLC NYC Taxi는 정형 대용량 일괄 처리의 증거다.

이 데이터셋은 크고, 정형이며, 이미 Parquet 형태로 다루기 좋다. 그래서 M2가 공통 실행 기반을 설명하기에 적합하다. 다만 지금 시점에서 Taxi의 목표를 잘못 잡으면 일정이 크게 흔들린다.

Taxi의 목표는 당장 전체 데이터를 완벽히 처리하는 것이 아니다. 한 달치 또는 작은 범위에서 시작해도 된다. 중요한 것은 같은 방식으로 더 큰 범위까지 확장할 수 있는 실행 기반과 경로 규칙을 확인하는 것이다. TLC NYC Taxi Dataset을 쓰는 이유는 예전 M2 계획을 이어받으면서도, AskLake가 정형 빅데이터 ETL을 수행할 수 있음을 보여주기 위해서다.

M2가 맡아야 할 것은 Spark 실행, MinIO 또는 S3 호환 저장소, Parquet 경로 규칙, SQL 실행 확인이다. Taxi의 비즈니스 변환 정책까지 M2가 정하면 M3와 책임이 겹친다. Taxi 결과를 카탈로그에 어떻게 연결할지도 M5와 맞춰야 한다.

프로젝트 관리자는 Taxi 작업을 볼 때 “대용량 전체 처리가 끝났는가”보다 “공통 실행 기반이 다른 데이터셋에도 재사용 가능하게 정리되고 있는가”를 봐야 한다.

## Kafka 쇼핑몰 데이터의 의미

Kafka 경로는 실시간형 수집의 증거다.

사용할 원본은 Kaggle의 Ecommerce Behavior Data from Multi Category Store다. 이 데이터는 실제 실시간 데이터 소스는 아니지만, 파일에 있는 이벤트를 Kafka 주제로 재생하면 실시간처럼 데이터가 들어오는 장면을 만들 수 있다.

여기서 중요한 것은 “진짜 운영용 실시간 시스템”을 만드는 것이 아니다. 발표에서 필요한 것은 원본 이벤트가 안정적으로 Kafka에 들어가고, 소비자가 그 이벤트를 읽을 수 있으며, 후속 처리 팀이 이해할 수 있는 원본 이벤트 계약이 남는 것이다.

M4는 어떤 주제 이름을 쓰는지, 이벤트 키를 무엇으로 삼는지, 이벤트 시간이 어떤 컬럼인지, 재생 명령이 무엇인지, 몇 건을 읽었는지, 실패하면 어떻게 보이는지 증거를 남겨야 한다.

M4가 Gold 결과 데이터 생성이나 카탈로그 저장까지 가져가면 안 된다. Kafka 경로의 1차 목표는 안정적인 원본 이벤트 수집이다. 그 이후의 정리와 변환은 M3, 저장과 조회는 M5의 책임이다.

## 모듈별 책임

### M1: 화면과 사용자 흐름

M1은 발표의 얼굴이다.

M1이 해야 할 일은 모든 내부 로직을 직접 구현하는 것이 아니다. 사용자가 실행 상태, 카탈로그, 질문 결과, 근거를 한 흐름으로 이해할 수 있게 보여주는 것이다.

화면에서 실행이 진행 중인지, 실패했는지, 성공했는지 보여야 한다. 카탈로그가 비어 있는지 갱신되었는지도 보여야 한다. 질문 응답이 어떤 데이터셋을 근거로 나왔는지도 드러나야 한다.

### M2: 공통 실행 기반

M2는 정형 데이터 담당자가 아니라 공통 실행 기반 담당자다.

Spark, 저장소 경로, Parquet 입출력, SQL 실행 확인이 M2의 핵심이다. Taxi는 이 실행 기반이 정형 대용량 데이터에 쓸 수 있음을 보여주는 좋은 예시일 뿐이다. M2 구현은 Taxi 전용 실행기가 아니라, 입력 형식과 경로만 바꾸면 다른 데이터셋에도 쓸 수 있는 공통 실행기여야 한다.

M2가 데이터 변환 의미, 카탈로그 저장 방식, 화면 표시까지 가져가면 책임이 커지고 충돌 가능성이 높아진다.

### M3: 데이터 구조 파악과 변환 정의

M3는 데이터의 의미를 읽고 분석 가능한 형태로 바꾸는 기준을 만든다.

Amazon Reviews JSON에서 어떤 필드가 있고, 어떤 타입이며, 어떤 값이 누락될 수 있는지 파악해야 한다. 그 다음 어떤 필드를 고르고, 어떤 값을 정리하고, 어떤 지표를 만들지 정의해야 한다.

M3가 가장 먼저 닫아야 하는 것은 Amazon Reviews JSON이다. Taxi와 Kafka까지 한 번에 처리하려고 하면 M3가 병목이 된다.

### M4: Kafka 수집

M4는 Kafka로 데이터가 들어오는 경로를 책임진다.

주제 이름, 이벤트 키, 이벤트 시간, 원본 메시지 형태, 재생 명령, 소비 확인 결과가 M4의 산출물이다. M4가 보여줘야 하는 것은 “이 데이터가 실시간처럼 들어올 수 있다”는 증거다.

M4가 데이터 정제, Gold 결과 생성, 카탈로그 저장까지 맡으면 안 된다.

### M5: 작업 실행과 카탈로그

M5는 실행과 기록의 중심이다.

작업 정의를 받아 실행하고, 실행 결과를 표준 형태로 남기고, 성공한 결과를 카탈로그에 저장해야 한다. 이때 실행 결과와 카탈로그가 같은 실행 번호나 데이터셋 맥락으로 이어져야 한다.

M5가 M3의 변환 로직을 내부에 복사하거나, M2의 실행 기반을 직접 새로 만들면 책임 경계가 깨진다.

### M6: 카탈로그 기반 질문 응답

M6는 사용자의 질문을 실제 데이터 맥락과 연결한다.

M6가 해야 할 일은 그럴듯한 문장을 만드는 것이 아니다. 카탈로그에 있는 데이터셋 설명, 스키마, 지표, 품질 정보, 계보를 보고 어떤 데이터셋을 사용할지 고르고, 안전한 질의와 답변 근거를 만드는 것이다.

M6가 고정된 예시 답변에 머물면 AskLake의 핵심 가치가 약해진다.

## 프로젝트 관리자가 매일 봐야 하는 것

프로젝트 관리자는 매일 핵심 발표 경로를 먼저 확인해야 한다.

Amazon Reviews JSON에서 시작해 M3, M5, M6, M1까지 실제로 이어지는지 본다. 어느 한 지점이 막히면 그 지점이 그날의 최우선 관리 대상이다.

그 다음에는 책임 중복을 확인한다. 같은 Spark 설정을 여러 명이 만들고 있지는 않은지, 출력 경로 규칙을 여러 곳에서 다르게 정하고 있지는 않은지, 카탈로그 저장을 M3와 M5가 동시에 만지고 있지는 않은지 봐야 한다.

그 다음에는 계약 변경을 확인한다. 작업 흐름 정의, 실행 결과, 카탈로그 정보, Kafka 이벤트 구조, 질문 응답 구조가 바뀌면 반드시 생산자와 소비자를 함께 확인해야 한다.

마지막으로 증거를 확인한다. “돌아간다”는 말만 있으면 부족하다. 실행 명령, 처리 건수, 출력 위치, 카탈로그 응답, 화면 캡처, 실패 로그 중 하나라도 남아야 한다.

## 가장 큰 위험: 범위 확대

이번 프로젝트에서 가장 큰 위험은 기능을 너무 많이 하려는 것이다.

Taxi 전체 데이터를 완벽히 처리하려는 순간, Kafka를 Gold 결과까지 연결하려는 순간, Spark와 Airflow와 카탈로그와 질문 응답을 한 번에 묶으려는 순간 일정이 흔들린다.

프로젝트 관리자는 기능을 줄이는 사람이 아니라 순서를 정하는 사람이어야 한다. 먼저 Amazon Reviews JSON 핵심 경로를 닫는다. 그 다음 Taxi로 정형 일괄 처리 증거를 붙인다. 그 다음 Kafka로 실시간형 수집 증거를 붙인다.

## 두 번째 위험: 책임 중복

책임이 겹치면 나중에 병합 충돌보다 더 큰 문제가 생긴다.

Spark 실행을 M2와 M3가 따로 만들면 하나를 버려야 한다. 출력 경로 규칙을 M2와 M5가 다르게 정하면 카탈로그가 무엇을 믿어야 하는지 흐려진다. 카탈로그 저장을 M3가 직접 해버리면 M5의 책임이 약해진다. 질의 안전장치를 M6 밖에서 따로 만들면 질문 응답의 책임 경계가 깨진다.

프로젝트 관리자는 누가 더 많이 구현했는지보다 같은 책임을 두 명이 구현하고 있지 않은지를 먼저 봐야 한다.

## 세 번째 위험: 계약 흔들림

이 프로젝트에는 여러 팀원이 함께 맞춰야 하는 계약이 있다.

대표적으로 원본 설정, 변환 정의, 실행 설정, 작업 흐름 정의, 실행 결과, 카탈로그 정보, Kafka 이벤트 계약, 질문 응답 결과가 있다. 코드 이름으로는 `SourceConfig`, `TransformSpec`, `RuntimeConfig`, `WorkflowDefinition`, `ExecutionResult`, `CatalogMetadata`, `KafkaTopicContract`, `AIQueryResult`에 해당한다.

이 계약들은 한 사람이 바꾸면 다른 사람이 영향을 받는다. M3가 변환 정의를 바꾸면 M5 실행이 영향을 받는다. M5가 카탈로그 구조를 바꾸면 M6와 M1이 영향을 받는다. M4가 Kafka 메시지 구조를 바꾸면 M3가 영향을 받는다.

계약 변경은 막을 일이 아니다. 다만 조용히 바뀌면 안 된다. 프로젝트 관리자는 계약 변경이 생겼을 때 반드시 생산자와 소비자가 같은 이해를 갖고 있는지 확인해야 한다.

## 네 번째 위험: 증거 부족

발표 직전에 가장 위험한 말은 “아마 돌아갈 거예요”다.

각 기능은 작아도 실행 증거가 있어야 한다. 어떤 명령으로 실행했는지, 몇 건을 처리했는지, 출력이 어디에 생겼는지, 카탈로그 응답이 어떤 모양인지, 화면에서 무엇이 보였는지 남겨야 한다.

코드가 많아도 증거가 없으면 완료가 아니다. 반대로 작은 기능이라도 실행 증거가 명확하면 다음 팀원이 믿고 이어갈 수 있다.

## 다섯 번째 위험: 데이터셋 접근성

Amazon Reviews JSON은 이미 기존 문서와 샘플, 테스트 맥락이 있어 기준으로 삼기 쉽다.

TLC NYC Taxi는 공개 데이터셋이고 Parquet 기반이라 정형 처리 증거로 쓰기 좋다.

하지만 Kaggle 쇼핑몰 데이터셋은 팀원이 실제로 받을 수 있는지, 사용 조건에 문제가 없는지, 원본 파일 형식이 무엇인지, 샘플 행 구조가 어떤지 확인해야 한다. 이 확인 없이 Kafka 작업을 시작하면 메시지는 보낼 수 있어도 후속 처리 팀이 받을 수 없는 형태가 될 수 있다.

## 이슈는 어떻게 쪼개야 하는가

좋은 이슈는 하루 안에 설명 가능한 결과물을 만든다.

M3가 Amazon Reviews JSON에서 스키마와 기본 지표를 뽑는 일은 좋은 이슈다. M5가 성공한 실행 결과를 카탈로그 최신 항목으로 연결하는 일도 좋은 이슈다. M4가 Kaggle 쇼핑몰 데이터를 Kafka 주제로 재생하는 확인 명령을 만드는 일도 좋은 이슈다. M2가 Spark 실행 확인 결과를 표준 실행 결과 형태로 반환하는 일도 좋은 이슈다.

반대로 “모든 데이터셋 처리 완성”, “Spark와 Kafka와 카탈로그와 질문 응답 전부 연결”, “Taxi 전체 적재부터 화면 표시까지 한 번에 구현” 같은 이슈는 너무 크다. 이런 이슈는 프로젝트 관리자가 쪼개야 한다.

## Pull Request를 볼 때의 기준

Pull Request를 볼 때 프로젝트 관리자는 코드 줄 수보다 흐름과 증거를 먼저 봐야 한다.

이 Pull Request가 어떤 이슈를 닫는지 확인한다. 핵심 발표 경로를 전진시키는지, 아니면 보조 증거를 보강하는지 확인한다. 계약을 바꿨다면 누가 영향을 받는지 확인한다. 실행 증거가 남았는지 확인한다.

테스트가 통과했다는 사실은 중요하지만 충분하지 않다. 그 테스트가 실제 사용자 흐름을 검증하는지도 봐야 한다. 자동 검사는 기본 안전망이지 프로젝트 관리자의 판단을 대체하지 않는다.

## 내일 아침에 먼저 확인할 것

내일 가장 먼저 확인할 것은 Kafka 데이터셋이다.

Kaggle 링크는 정해졌다. 이제 M4가 실제 파일을 받을 수 있는지, 파일 크기와 형식이 무엇인지, 샘플 행이 어떤 구조인지, 이벤트 키와 이벤트 시간을 무엇으로 삼을지 확인해야 한다. 이 질문이 답되지 않으면 Kafka 작업은 겉으로는 진행되는 것처럼 보여도 M3와 M5가 이어받기 어렵다.

그 다음은 M2 범위 확인이다. M2는 공통 실행 기반을 제공해야 한다. Taxi는 그 실행 기반을 보여주는 예시다. Taxi 전체 변환 정책이나 카탈로그 연결까지 M2가 혼자 가져가면 안 된다.

동시에 M3는 Amazon Reviews JSON을 먼저 닫아야 한다. M3에게 지금 물어야 할 질문은 “모든 데이터 형식을 처리할 수 있느냐”가 아니다. “Amazon Reviews JSON에서 M5와 M6가 쓸 수 있는 최소 구조와 변환 정의를 언제 줄 수 있느냐”다.

M5와 M6는 카탈로그 구조를 맞춰야 한다. M6가 실제 카탈로그 기반 답변을 하려면 M5가 어떤 정보를 저장해야 하는지 합의해야 한다.

M1은 실행 흐름과 근거가 보이게 해야 한다. 화면이 예쁜지도 중요하지만, 지금 더 중요한 것은 사용자가 데이터 입력, 실행, 카탈로그, 질문 응답, 근거를 따라갈 수 있느냐이다.

## 최종 운영 원칙

이번 주 프로젝트 관리는 한 문장으로 정리된다.

**Amazon Reviews JSON 핵심 경로를 먼저 닫고, Taxi와 Kafka는 확장 가능성을 보여주는 증거로 붙인다.**

이 원칙을 지키면 여섯 명이 각자 다른 일을 해도 하나의 발표 흐름으로 모일 수 있다. 이 원칙이 흔들리면 각자 열심히 만들었는데 마지막에 연결되지 않는 상황이 생긴다.

프로젝트 관리자가 매일 팀에 던져야 할 질문은 이것이다.

**오늘 우리가 만든 것이 AskLake의 신뢰 루프를 더 닫았는가?**

이 질문에 답하지 못하는 작업은 중요해 보여도 우선순위가 낮다. 반대로 작아 보여도 데이터 입력, 실행, 카탈로그, 질문 응답, 화면 근거 중 하나를 확실히 연결하는 작업은 이번 주에 가치가 크다.
