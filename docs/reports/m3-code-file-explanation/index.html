M3 code explanation board

M3 Logical L0-L16 코드 흐름과 산출물 책임을 한 화면에서 보는 보드

이 보드는 M3가 맡는 Transform 계약 생성 범위를 설명한다. 원본 보존부터 Bronze/Silver/Gold 추천, 사용자 승인, preview-only spec, 품질 gate, catalog/query/vector handoff까지 빠뜨리지 않고 연결한다. Spark/MinIO는 검증 하네스이며, production 실행과 저장은 M2/M5 경계다.

M3 scope: recommendation / spec / catalog handoff AI: bounded control-plane only Runtime execution: M2 boundary Catalog / workflow storage: M5 boundary Query / semantic context: M6 boundary

쉬운 말로 보는 M3 전체 동작

M3는 raw 데이터를 직접 크게 돌려서 결과 테이블을 확정하는 파트가 아니다. M3는 unknown data가 들어왔을 때 원본을 잃지 않고, 어떤 정제를 거쳐 Silver가 되어야 하는지, 어떤 근거가 있을 때만 Gold를 만들 수 있는지, 그리고 그 결정을 M2/M5/M6가 읽을 수 있는 계약으로 바꾸는 파트다.

실제 흐름은 먼저 원본을 보존하는 데서 시작한다. L0는 파일을 바꾸지 않고 위치표와 지문을 만들고, L1은 제한된 sample을 Bronze 봉투로 감싸며, L2는 그 sample을 보고 format과 schema/profile snapshot을 만든다. 이 구간은 “데이터를 이해한다”기보다 “나중에 같은 데이터를 다시 찾고 같은 근거로 판단할 수 있게 만든다”에 가깝다.

예를 들어 Amazon 리뷰 JSONL이 들어오면 M3는 처음부터 `이건 상품 건강도 Gold다`라고 확정하지 않는다. 먼저 `asin`이 상품 id처럼 보이는지, `overall`이 평점처럼 보이는지, `reviewText`가 리뷰 본문인지 profile로 확인한다. 그 다음 review_count나 average_rating은 만들 수 있는지 보고, view_count나 purchase_count처럼 리뷰 파일에 없는 값은 없다고 표시한다. 없는 값을 0으로 채워 Gold를 꾸미지 않는 것이 핵심이다.

택시 CSV/Parquet가 들어오면 다른 판단을 한다. `pickup_datetime`, `fare_amount`, `passenger_count` 같은 field는 product_health가 아니라 시간/지역/요금 집계 Gold 후보가 될 수 있다. 그래서 같은 L7 Gold 단계라도 데이터가 리뷰인지 택시인지에 따라 추천 metric과 risk_score component가 달라져야 한다. M3는 고정 공식을 우겨 넣는 것이 아니라, 실제 field 근거를 보고 만들 수 있는 Gold 방법을 제안한다.

그 다음 AI는 전체 데이터를 실시간으로 보는 것이 아니라, L2에서 압축된 profile evidence만 본다. L3는 AI에게 넘겨도 되는 근거만 골라 PII와 raw payload 노출을 줄이고, L4/L5는 기존 metadata나 template을 참고하되 실제 source field 근거가 있는지 다시 대조한다. 그래서 AI 추천은 그럴듯한 말이 아니라 “어떤 field와 어떤 evidence 때문에 이 Silver/Gold 후보가 가능한지”를 남겨야 한다.

Silver와 Gold는 서로 다른 결정이다. Silver는 Bronze를 깨끗하고 query 가능한 형태로 정리하는 정책이고, Gold는 그 Silver 위에서 product_health, risk_score, aggregate 같은 의미 지표를 만드는 정책이다. Gold에 필요한 view_count, purchase_count, delivery_count 같은 근거가 없으면 0으로 꾸미지 않고 missing/deferred로 남긴다. risk_score도 하나의 고정 수식으로 박지 않고, 데이터에 실제로 있는 component를 보고 추천한 뒤 사용자가 승인하게 한다.

사용자가 L9에서 승인해야만 L10/L11이 preview_only spec을 만든다. 그 spec은 M2가 Spark 같은 실행 엔진으로 돌릴 수 있는 설계도이고, M3가 production write를 직접 하지 않는다. preview 결과가 돌아오면 L13-L15가 품질, catalog 안전성, Gold 준비도를 나눠 판정하고, L16이 M5/M6가 읽을 catalog/query/vector handoff package로 묶는다.

입력사용자가 등록한 모르는 구조의 CSV/JSON/JSONL/Parquet/text와 source_id/run_id. M3는 처음부터 schema를 안다고 가정하지 않는다.
AI가 보는 것원본 전체 row가 아니라 profile snapshot, field hint, 제한된 example, retrieval candidate, missing evidence다. 대용량/실시간 상황에서 per-row AI는 금지한다.
사용자가 고르는 것Silver 정제 정책, Gold 생성 여부, product-health metric, risk_score component/weight, vector handoff 여부, Gold-to-Gold 파생 여부다.
M2로 가는 것preview_only transform spec과 source_unit/artifact_ref다. 실제 Spark session, shuffle, output write, MinIO 적재는 M2 경계로 둔다.
M5/M6로 가는 것approval state, catalog sync package, SQL context, allowed columns, caveat, artifact_reference_manifest, semantic/vector handoff template이다.

각 단계별로 그래서 어떻게 되는지

이 표는 코드 이름이나 계약 이름을 먼저 외우라는 표가 아니다. 사용자가 데이터를 넣으면 각 L단계가 실제로 무엇을 보고, 무엇을 만들고, 그 결과 다음 단계가 어떻게 움직이는지 보는 표다.

읽는 법: 왼쪽 L번호를 보고, 가운데의 “들어오면 -> M3가 해서 -> 그래서 남는 것 -> 다음에 어떻게 됨”만 따라가면 된다. 파일명과 코드명은 나중에 상세 코드 화면에서 확인하면 된다.

L0

원본 파일에 신분증을 붙인다

파일을 고치기 전에 “이 파일이 누구인지”부터 고정한다.

사용자가 어떤 파일을 넣었는지 모르는 상태에서는 내용 분석보다 원본 추적이 먼저다. L0가 끝나면 “이 원본은 어디 있고, 나중에 같은 파일인지 어떻게 확인할지”가 정해진다.

들어오면파일 또는 폴더 경로, source_id, run_id가 들어온다.
M3가 해서파일마다 object_id와 source_unit_id를 붙이고 위치, 크기, checksum을 적는다.
그래서 남는 것원본 파일 신분증과 다시 찾는 위치표가 남는다.
다음에 어떻게 됨L1이 이 위치표를 보고 sample을 뽑는다. M2도 나중에 같은 원본을 읽을 수 있다.

예시: Amazon 리뷰 파일이 들어오면 리뷰 내용은 아직 보지 않고, “이 파일은 F드라이브 어디에 있고 크기는 얼마다”부터 기록한다.

상세 코드
L1

원본 일부를 샘플 봉투에 담는다

전체를 다 읽지 않고 대표 조각과 실패 조각을 같이 남긴다.

L1은 큰 파일 전체를 처리하지 않는다. 일부 record만 꺼내되, 그 record가 원본의 어느 위치에서 왔는지 같이 저장한다. 깨진 줄도 버리지 않고 따로 모아 둔다.

들어오면L0의 파일 위치표와 source_unit_id가 들어온다.
M3가 해서정해진 row/byte 한도만 읽고 정상 sample과 실패 sample을 나눈다.
그래서 남는 것대표 sample, parse 실패 목록, 각 record의 원본 위치가 남는다.
다음에 어떻게 됨L2가 이 sample로 컬럼 모양을 추정하고, M2는 같은 규칙으로 전체를 처리한다.

예시: JSONL 100GB 중 일부 줄만 보되, 깨진 JSON 줄은 “실패”로 남겨 나중에 격리할 수 있게 한다.

상세 코드
L2

컬럼 건강검진표를 만든다

이 데이터가 어떤 모양인지 처음으로 정리한다.

L2는 sample을 보고 각 컬럼이 숫자인지, 날짜인지, 긴 글인지, 비어 있는 값이 많은지 본다. 아직 확정 정답이 아니라 “현재 sample 기준으로 이렇게 보인다”는 검사표다.

들어오면L1 sample과 실패 기록이 들어온다.
M3가 해서format 후보, 컬럼별 타입 후보, 빈값 비율, 예시 값을 계산한다.
그래서 남는 것데이터 모양표와 schema 지문이 남는다.
다음에 어떻게 됨L3가 이 표를 줄여 AI에게 보여주고, L6/L7 추천의 근거가 된다.

예시: `overall`은 평점 숫자 후보, `reviewText`는 리뷰 본문 후보, `reviewTime`은 날짜 후보로 표시된다.

상세 코드
L3

AI에게 줄 안전 요약본을 만든다

AI가 원본 전체를 보지 않게 막는 단계다.

L3는 AI에게 필요한 근거만 작게 포장한다. 원본 row 전체, 긴 본문 전체, 민감한 값은 그대로 넘기지 않는다. 그래서 대용량이나 실시간 데이터에서도 “모든 row를 AI가 본다”는 거짓 구조를 피한다.

들어오면L2의 컬럼 건강검진표와 제한된 예시가 들어온다.
M3가 해서PII 후보를 가리고 AI가 볼 정보량을 줄인다.
그래서 남는 것AI용 안전 요약본과 가림표가 남는다.
다음에 어떻게 됨L4~L8이 이 요약본으로 추천을 만든다. 원본 전체는 계속 M3 밖에 있다.

예시: 이메일 같은 값은 그대로 보여주지 않고 “민감 정보 후보 field”라고 표시한다.

상세 코드
L4

비슷한 과거 템플릿을 찾아온다

혼자 상상하지 않고 기존 catalog/schema 기억을 참고한다.

L4는 지금 데이터와 비슷한 schema, catalog 설명, Gold 템플릿 후보를 찾아온다. 단, 검색 결과는 정답이 아니라 “참고 후보”다.

들어오면AI 안전 요약본과 컬럼 특징이 들어온다.
M3가 해서비슷한 metadata, product-health 템플릿, vector 후보를 찾는다.
그래서 남는 것참고할 템플릿 후보와 왜 비슷한지 근거가 남는다.
다음에 어떻게 됨L5가 이 후보가 실제 데이터 재료를 갖는지 다시 확인한다.

예시: `asin`, `rating`, `reviewText`가 보이면 product health 템플릿을 참고 후보로 가져올 수 있다.

상세 코드
L5

추천 후보에 실제 재료가 있는지 확인한다

없는 지표를 만들어내지 않게 막는다.

L5는 템플릿이 요구하는 재료가 실제 컬럼에 있는지 확인한다. 이 단계가 있어야 review_count는 만들 수 있지만 conversion_rate는 못 만든다는 식의 구분이 가능하다.

들어오면템플릿 후보와 source field 후보가 들어온다.
M3가 해서상품 id, 리뷰, 평점, 조회, 구매, 배송 같은 재료 존재 여부를 대조한다.
그래서 남는 것가능한 지표와 근거 부족 지표 목록이 남는다.
다음에 어떻게 됨L7이 근거 있는 Gold만 추천하고, 없는 것은 보류로 표시한다.

예시: Amazon 리뷰만 있으면 `purchase_count`가 없으므로 conversion_rate는 “근거 부족”으로 남긴다.

상세 코드
L6

Silver 청소 추천서를 만든다

분석 전에 데이터를 어떻게 정리할지 제안한다.

L6는 어떤 컬럼을 남길지, 이름을 바꿀지, 숫자/날짜로 고칠지, 민감 정보는 숨길지 추천한다. 아직 실행이 아니라 사용자가 고칠 수 있는 추천서다.

들어오면컬럼 건강검진표와 AI 안전 근거가 들어온다.
M3가 해서select, cast, rename, mask, hash, quarantine 후보를 만든다.
그래서 남는 것Silver 정제 추천서가 남는다.
다음에 어떻게 됨L9에서 사용자가 승인해야 L10 작업 지시서로 내려간다.

예시: `overall`은 숫자 평점으로 바꾸고, `reviewerName`은 query에서 숨기라고 추천할 수 있다.

상세 코드
L7

Gold 표 후보와 risk_score 정책을 제안한다

의미 있는 분석용 표를 만들 수 있는지 본다.

L7은 Gold를 무조건 만들지 않는다. product health, 집계표, risk_score 같은 후보를 만들되, 실제 재료가 있는 컬럼만 쓴다. 공식과 가중치도 고정하지 않고 추천 상태로 둔다.

들어오면L5의 근거 확인 결과와 Silver 후보가 들어온다.
M3가 해서가능한 Gold 컬럼, 보류할 컬럼, risk_score 재료를 나눈다.
그래서 남는 것Gold 추천서와 risk_score 정책 초안이 남는다.
다음에 어떻게 됨L9에서 사용자가 Gold를 만들지, 보류할지, 공식을 고칠지 결정한다.

예시: 리뷰 데이터는 review_count와 average_rating은 가능하지만, 조회/구매 데이터가 없으면 conversion_rate는 보류한다.

상세 코드
L8

나중에 검색에 쓸 설명 후보를 만든다

벡터 검색은 직접 만들지 않고 넘길 준비만 한다.

L8은 텍스트 컬럼, catalog 설명, metric 설명 중에서 나중에 semantic search에 넣을 만한 후보를 정한다. M3가 embedding을 직접 만들지는 않는다.

들어오면텍스트 field 후보와 catalog/Gold 설명 후보가 들어온다.
M3가 해서검색에 넣을 정보와 넣으면 안 되는 민감 정보를 나눈다.
그래서 남는 것vector handoff 후보 문서가 남는다.
다음에 어떻게 됨M6나 extension이 semantic search를 만들 때 이 문서를 참고한다.

예시: 리뷰 본문은 검색 후보가 될 수 있지만, 사용자 이메일 같은 값은 제외되어야 한다.

상세 코드
L9

사용자 선택을 공식 결정으로 저장한다

AI 추천이 실행으로 넘어가기 전 사람이 확정한다.

L9는 Silver, Gold, risk_score, vector, Gold-to-Gold 선택 상태를 저장한다. 승인되지 않은 추천은 뒤 작업 지시서로 내려가지 않는다.

들어오면L6~L8 추천서와 사용자의 수정/승인 상태가 들어온다.
M3가 해서approved, deferred, rejected, not_requested 상태로 확정한다.
그래서 남는 것사용자 승인서와 결정 기록이 남는다.
다음에 어떻게 됨L10/L11은 여기서 승인된 것만 작업 지시서로 바꾼다.

예시: 사용자가 Gold를 아직 안 만들겠다고 하면 Silver는 진행하고 Gold는 not_requested로 남긴다.

상세 코드
L10

Silver 작업 지시서를 만든다

M2가 실행할 수 있는 형태로 바꾼다.

L10은 승인된 Silver 결정만 읽고 “어떤 컬럼을 선택하고, 어떻게 변환하고, 무엇을 숨길지”를 작업 지시서로 만든다. 실행 모드는 미리보기 전용이다.

들어오면L9에서 승인된 Silver 결정이 들어온다.
M3가 해서select, cast, mask, quarantine 같은 작업 순서를 정리한다.
그래서 남는 것Silver 미리보기 작업 지시서가 남는다.
다음에 어떻게 됨M2가 이 지시서를 Spark/local runner로 실행할 수 있다.

예시: “reviewTime을 날짜로 바꾸고 reviewerName은 숨겨라”가 실행 가능한 지시서가 된다.

상세 코드
L11

Gold 작업 지시서를 만든다

승인된 Gold 지표만 집계 방법으로 바꾼다.

L11은 Gold가 승인된 경우에만 지표 계산 방법을 만든다. 보류된 지표나 근거 없는 지표는 실행 지시서에 넣지 않는다.

들어오면L9의 Gold 승인 상태와 L7 Gold 추천서가 들어온다.
M3가 해서group_by, metric, denominator rule, risk_score policy를 정리한다.
그래서 남는 것Gold 미리보기 작업 지시서가 남는다.
다음에 어떻게 됨M2가 Silver 결과를 입력으로 Gold preview를 만들 수 있다.

예시: product_id별 review_count와 average_rating은 만들고, view_count가 없으면 conversion_rate는 제외한다.

상세 코드
L12

작업 지시서에 위험한 명령이 있는지 막는다

실행 전에 안전 검사를 한다.

L12는 Silver/Gold 작업 지시서가 안전한지 본다. 모든 row마다 AI 호출, production write, 임의 코드 실행 같은 것은 막는다.

들어오면L10/L11 작업 지시서가 들어온다.
M3가 해서허용된 작업인지, preview_only인지, 참조가 맞는지 검사한다.
그래서 남는 것통과/차단 결과와 차단 이유가 남는다.
다음에 어떻게 됨통과한 지시서만 M2 preview 실행 대상으로 볼 수 있다.

예시: “production 테이블에 바로 저장” 같은 명령이 들어오면 block된다.

상세 코드
L13

Silver 미리 실행 결과를 검사한다

정제가 실제로 쓸 만한지 본다.

L13은 M2가 미리 돌려 준 Silver 결과를 보고 row가 사라졌는지, 변환 실패가 많은지, 민감 정보가 노출됐는지 확인한다.

들어오면M2의 Silver preview 실행 결과가 들어온다.
M3가 해서row_count, cast 실패, quarantine, PII/query 노출을 검사한다.
그래서 남는 것Silver 품질 검사표와 노출 위험 보고서가 남는다.
다음에 어떻게 됨L15가 Silver를 ready/warn/block으로 판단할 근거로 쓴다.

예시: 평점 숫자 변환 실패가 너무 많으면 warn이나 block 근거가 된다.

상세 코드
L14

Gold 지표가 의미적으로 맞는지 검사한다

Gold 문제를 Silver 문제와 섞지 않는다.

L14는 Gold 지표가 실제 근거와 공식을 갖는지 본다. Gold가 보류되어도 그 이유를 남기고, Silver 자체를 실패로 만들지는 않는다.

들어오면Gold 추천/승인 상태와 preview 근거가 들어온다.
M3가 해서metric별 분자/분모, formula, missing evidence, caveat를 확인한다.
그래서 남는 것Gold 준비도 보고서와 의미상 주의 문구가 남는다.
다음에 어떻게 됨L15가 Gold를 available/deferred/not_ready로 나눌 수 있다.

예시: 배송 데이터가 없으면 late_delivery_rate는 Gold 지표에서 보류된다.

상세 코드
L15

최종 사용 가능 상태를 세 갈래로 판정한다

Silver 품질, 노출 안전, Gold 준비도를 따로 본다.

L15는 처리 품질, catalog/query 안전성, Gold 준비도를 분리해 판단한다. Gold가 보류되어도 Silver가 안전하면 M6가 Silver context를 쓸 수 있다.

들어오면L13 Silver 검사와 L14 Gold 검사 결과가 들어온다.
M3가 해서processing_quality, catalog_safety, gold_readiness를 각각 pass/warn/block으로 본다.
그래서 남는 것M6가 읽을 context 상태표가 남는다.
다음에 어떻게 됨L16이 이 상태표를 catalog와 query package에 그대로 넣는다.

예시: Silver는 ready지만 Gold는 deferred일 수 있다. 이 둘을 하나로 뭉개지 않는다.

상세 코드
L16

M5/M6가 읽을 최종 전달 꾸러미를 만든다

모든 결정과 검사 결과를 한 패키지로 묶는다.

L16은 catalog, SQL context, vector handoff, artifact 목차를 묶는다. 다른 M이 어떤 table/column/metric을 써도 되는지, 어떤 caveat를 붙여야 하는지 여기서 읽는다.

들어오면앞 단계의 결정, 작업 지시서, 품질 판정, artifact 목록이 들어온다.
M3가 해서M5/M6가 읽을 catalog package와 query context를 만든다.
그래서 남는 것최종 전달 꾸러미와 artifact 목차가 남는다.
다음에 어떻게 됨M5는 저장/워크플로우에 쓰고, M6는 질문 답변 context에 쓴다.

예시: M6는 여기서 “이 컬럼은 query 가능, 이 Gold metric은 caveat 필요” 같은 정보를 읽는다.

상세 코드

전체 흐름도

위쪽은 큰 묶음, 아래쪽은 실제 L0~L16 단계다. 각 카드를 누르면 해당 단계의 코드 설명으로 이동한다. 먼저 “무엇이 남는지”를 보고, 필요할 때만 코드 파일명을 보면 된다.

L0-L2 먼저 확인원본 위치, 일부 sample, 컬럼 상태표를 만든다. 아직 정제나 Gold를 확정하지 않는다.
L3-L8 추천 만들기AI가 볼 요약본을 만들고, Silver 청소법과 Gold 후보를 제안한다. 원본 전체를 AI에게 넘기지 않는다.
L9 사용자가 고르기추천을 그대로 쓸지, 고칠지, 보류할지, 거절할지 기록한다.
L10-L12 작업 지시서승인된 것만 M2가 미리 실행할 수 있는 지시서로 만들고 위험한 명령은 막는다.
L13-L15 결과 검사미리 실행한 결과가 쓸 만한지, 민감 정보가 안전한지, Gold가 준비됐는지 따로 본다.
L16 다른 M에 전달M5가 저장하고 M6가 질문 답변에 쓸 최종 꾸러미를 만든다.

tools/m3_contract/l10_handoff.py남는 것: 저장/질의/검색용 최종 패키지

산출물 기준 흐름: M3가 남기는 문서들이 어떤 순서로 생기는지

여기서는 Raw, Bronze, Silver 같은 내부 용어보다 먼저 “M3가 실제로 남기는 문서가 무엇이고, 그 문서를 누가 읽고, 다음 문서가 왜 필요한지”를 쉬운 이름으로 설명한다. 파일명은 아래에 작게 붙였지만, 먼저 봐야 할 것은 역할이다.

한 문장으로 말하면

M3는 데이터를 바로 고치는 사람이 아니라, 데이터가 어디서 왔는지 적은 보관증부터 정제 추천서, 사용자의 승인서, M2가 실행할 작업 지시서, M5/M6가 읽을 최종 전달 꾸러미까지 차례로 만드는 역할이다.

왜 이렇게 나누는가

모르는 구조의 큰 데이터는 처음부터 “정답 Gold”를 만들 수 없다. 먼저 원본을 잃지 않고, 일부를 보고, 모양을 검사하고, AI 추천을 만들고, 사용자가 고친 뒤, 미리 실행해 본 결과까지 확인해야 안전하게 다른 M에게 넘길 수 있다.

1 파일 신분증

원본 파일을 바꾸지 않고, 어디에 있는지와 나중에 같은 파일인지 확인할 지문을 적는다.

L0: source_manifest, object_stream_manifest
2 샘플 봉투

원본 전체가 아니라 일부 record를 꺼내고, 어느 파일의 어느 위치에서 왔는지 같이 담는다.

L1: sample, rescue lane
3 컬럼 건강검진표

각 컬럼이 숫자인지, 날짜인지, 텍스트인지, 비어 있는 값이 많은지, 위험한 정보 후보가 있는지 적는다.

L2: profile, schema fingerprint
4 AI에게 줄 안전 요약본

원본 전체 대신 작은 근거만 AI에게 넘긴다. 민감한 값이나 너무 긴 원문은 줄이거나 가린다.

L3-L5: AI input, template 근거 확인
5 정제/Gold 추천서

어떤 컬럼을 남기고 고칠지, 어떤 Gold 표를 만들 수 있는지, 없는 지표는 왜 보류해야 하는지 쓴다.

L6-L8: Silver 추천, Gold 추천, vector 후보
6 사용자 승인서

AI 추천을 그대로 실행하지 않고, 사용자가 승인/수정/보류/거절한 선택을 공식 기록으로 남긴다.

L9: approval_state, policy decision
7 작업 지시서와 미리 실행 검사표

M2가 실행할 수 있는 preview 전용 지시서를 만들고, 실행 결과가 쓸 만한지 품질과 노출 위험을 검사한다.

L10-L15: transform spec, validation, gate
8 최종 전달 꾸러미

M5가 저장하고 M6가 질문 답변에 쓸 수 있도록 catalog, SQL context, caveat, artifact 목차를 묶는다.

L16: catalog package, SQL context, artifact resolver
Amazon 리뷰 데이터 예시

리뷰 파일에서 상품 id와 평점, 리뷰 본문은 찾을 수 있다. 그래서 review_count, average_rating, negative_review_rate 같은 Gold 후보는 만들 수 있다. 하지만 view_count나 purchase_count가 파일에 없으면 conversion_rate는 만들지 않고 보류 사유를 남긴다.

택시 데이터 예시

택시 데이터는 상품 건강도보다 시간대별 운행 수, 지역별 요금, 이상치 탐지 같은 Gold 후보가 더 자연스럽다. 같은 흐름을 쓰지만, 추천되는 Gold 표와 risk_score 재료는 데이터의 실제 컬럼에 맞게 달라진다.

원본을 잃지 않기
파일 신분증파일을 고치지 않고 “이 파일은 어디 있고, 크기는 얼마고, 나중에 같은 파일인지 어떻게 확인할지”를 적는다. 주민등록증처럼 원본을 식별하는 문서다.
샘플 봉투전체를 다 열지 않고 일부만 꺼낸다. 꺼낸 줄마다 “어느 파일의 몇 번째 줄에서 왔는지”를 붙여서, 문제가 생기면 원본 위치로 돌아갈 수 있게 한다.
다음에 쓰는 곳M2가 전체 데이터를 처리할 때 이 위치표를 보고 같은 원본을 읽는다. M3는 원본을 복사하거나 직접 크게 실행하지 않는다.
모양을 알아보기
컬럼 건강검진표각 컬럼이 숫자인지, 날짜인지, 텍스트인지, 비어 있는 값이 많은지 본다. 예를 들면 `overall`은 평점 후보, `reviewText`는 긴 글 후보처럼 표시한다.
AI 안전 요약본AI에게 원본 전체를 주지 않는다. 컬럼 이름, 타입 후보, 비어 있는 비율, 제한된 예시만 주고 민감한 값은 가린다.
다음에 쓰는 곳이 요약본이 있어야 AI가 Silver 청소법과 Gold 후보를 추천한다. 모든 row를 AI가 실시간으로 보는 구조가 아니다.
청소법 정하기
Silver 청소 추천서어떤 컬럼을 남길지, 이름을 바꿀지, 숫자로 바꿀지, 날짜로 바꿀지, 민감 정보는 숨길지 추천한다.
사용자 확인AI 추천을 바로 실행하지 않는다. 사용자가 “이 컬럼은 남겨라”, “이건 숨겨라”, “이건 버리지 말고 격리해라”처럼 고칠 수 있다.
다음에 쓰는 곳승인된 청소법만 M2가 실행할 작업 지시서로 바뀐다. 승인 전에는 아직 제안서일 뿐이다.
의미 표 만들지 고르기
Gold 추천서상품별 건강도, 시간대별 요약, 위험 점수 같은 “분석용 표” 후보를 만든다. 단, 실제 컬럼 근거가 있는 지표만 후보가 된다.
없는 재료 표시리뷰 파일에 조회수나 구매수가 없으면 conversion_rate는 만들지 않는다. 없는 값을 0으로 채우는 대신 “근거 없음” 또는 “보류”로 남긴다.
사용자 확인risk_score 공식과 가중치도 고정으로 박지 않는다. 데이터에 있는 재료를 보고 추천하고, 사용자가 승인해야 한다.
넘기기 전에 검사하기
작업 지시서사용자가 승인한 내용만 M2가 실행할 수 있는 preview 전용 지시서로 바꾼다. production 저장 명령은 넣지 않는다.
미리 실행 검사표M2가 미리 돌린 결과를 보고 row가 사라졌는지, 변환 실패가 많은지, 민감 정보가 노출됐는지 검사한다.
최종 전달 꾸러미M5는 저장과 workflow에 쓰고, M6는 질문 답변에 쓸 수 있는 table/column/metric과 주의 문구를 읽는다.

코드 읽기 묶음

L0-L16을 한 줄로만 보면 길다. 실제 설명과 리뷰는 아래 7개 묶음으로 읽는 것이 빠르다.

L0-L2 Ingest Evidence원본 pointer, Bronze sample, profile/schema snapshot. AI나 Spark production 실행이 아니라 M3 control-plane evidence다.
L3-L5 AI Input SafetyAI-safe input pack, metadata retrieval, candidate grounding. raw full scan과 row-level AI를 막는 구간이다.
L6-L8 Draft RecommendationSilver draft, Gold draft, vector handoff draft. 아직 실행 spec이 아니라 사용자가 편집할 추천 초안이다.
L9 DecisionSilver/Gold/Gold-to-Gold/risk/vector 선택 상태를 분리한다. Gold 미요청이 Silver를 막지 않는다.
L10-L12 Compilerpreview-only deterministic spec과 compiler validation. M2가 실행할 수 있는 계약으로만 내린다.
L13-L15 Evidence Gatepreview evidence와 3-axis readiness. 실행 품질, catalog safety, Gold readiness를 섞지 않는다.
L16 HandoffM5 catalog storage와 M6 SQL/vector context가 읽을 package를 만든다.
