# M3 v2.1.1 L0-L10 계약 구현 PR 보고서

## 1. 작업 범위

이번 변경은 M3가 맡는 unknown CSV/JSON/JSONL 입력 분석, AI-safe recommendation, deterministic preview spec, catalog/query handoff 계약을 `v2.1.1` 기준으로 구현한 PR 준비 작업이다.

M3 core에 포함한 범위는 다음과 같다.

- `L0` raw object/source unit manifest, replay pointer, checksum/prefix checksum policy
- `L1` Bronze envelope sample manifest, rescue lane, object-backed replay locator
- `L2` CSV/JSON/JSONL profile snapshot, schema fingerprint, format-specific profile
- `L3` raw payload를 AI에 넘기지 않는 redaction-first AI evidence pack
- `L4` Silver policy recommendation draft, Gold model recommendation draft, AI generation trace
- `L5` Silver/Gold/Gold-to-Gold decision state, approval_state, recommendation diff
- `L6` preview-only Silver/Gold deterministic spec compiler, operation params, compiler validation
- `L7` Silver preview validation import contract, PII exposure report, Silver quality axis
- `L8` Gold preview input/readiness report, metric definition draft, semantic caveat report
- `L9` processing/catalog/gold three-axis gate and M6 context status
- `L10` catalog metadata draft, SQL context pack, artifact reference manifest, catalog sync package, compatibility exports

M3 core에 넣지 않은 범위는 다음과 같다.

- Spark production execution ownership
- MinIO/S3 production write ownership
- M5 catalog persistence 실행
- M6 SQL runtime 실행
- streaming runtime, watermark, checkpoint 운영
- row-level realtime AI cleaning

이 항목들은 로컬 검증 하네스나 extension hook으로만 남기고, M3 계약 산출물에는 "M2/M5/M6가 실행 또는 저장한다"는 handoff 상태만 기록한다.

### 2.2 v2.1.1 P0/P1 반영 사항

구현에서 직접 닫은 계약 항목:

- `source_unit_id`를 L0 object와 L1/L2/L6 preview scope까지 전달한다.
- Gold가 `deferred` 또는 `not_requested`여도 `gold_generation_spec`, `gold_preview`, `metric_definition_draft`, `gold_readiness_axis` artifact를 생성한다.
- `gold_readiness_axis_ref`는 nullable이 아니다.
- L9에서 Silver context와 Gold readiness를 분리한다. Gold가 deferred여도 Silver는 processing/catalog 기준으로 ready 또는 ready_with_caveat가 될 수 있다.
- L10 package의 top-level `m6_context_status`가 L9 `gate_summary`와 L10 `sql_context_pack`과 일치해야 한다.
- L6 spec은 `write_mode=preview_only`만 생성한다.
- L6 `preview_scope`는 `source_unit_ids[]`, `object_ids[]`, `stream_window_ids[]`만 사용하고 legacy `window_id`는 생성하지 않는다.
- Gold aggregate params는 `input_ref`, `group_by[]`, `dimensions[]`, `measures[]`, `time_window`, `cardinality_guard` 구조를 사용한다.
- PII 처리는 `pii_handling=none|mask|hash`와 `catalog_exposure`, `query_context_exposure`로 분리한다.
- `*_ref`가 실제 artifact id로 해석되도록 `artifact_reference_manifest.json`를 생성한다.

### 2.3 Artifact Reference Manifest 보강

초기 구현에서는 L10 `artifact_reference_manifest.json`가 자기 자신과 마지막 `handoff_package.json`를 누락할 수 있었다. 이를 2-pass 생성으로 보강했다.

현재 동작:

1. L10 catalog/sql/lineage/package를 먼저 쓴다.
2. 1차 artifact manifest를 쓴다.
3. handoff package를 쓴다.
4. 2차 artifact manifest를 다시 생성해 handoff package와 manifest artifact id까지 포함한다.

이 방식은 downstream에서 `artifact_reference_manifest_ref`, `catalog_sync_contract_package_ref`, `sql_context_ref`를 모두 artifact id로 resolve할 수 있게 하기 위한 것이다.

## 3. 현재 프로젝트 계약과 연결

`exports/`에는 main에 이미 들어간 Week2 계약과 연결 가능한 호환 산출물을 함께 만든다.

- `exports/transform_spec.json`: `contracts/transform_spec.sample.json` 계열 소비자와 연결
- `exports/schema_definition.json`: `contracts/schema_definition.sample.json` 계열 소비자와 연결
- `exports/workflow_definition.json`: `contracts/workflow_definition.sample.json` 계열 소비자와 연결
- `exports/catalog_metadata.json`: `contracts/catalog_metadata.sample.json` 계열 소비자와 연결

이 export는 M3가 production 실행을 맡는다는 뜻이 아니다. M3는 추천과 spec/handoff를 만들고, M5 runner selection 및 M2 execution이 실제 실행을 맡는 구조를 유지한다.

## 4. 검증 결과

### 4.1 M3 계약 단위 테스트

실행:

```powershell
python -m pytest tests\test_m3_expanded_contract.py -q
```

```text

```

검증 내용:

- deferred/not_requested Gold에서도 Silver context가 Gold readiness에 오염되지 않는지 확인
- L9 `gold_readiness_axis.json`가 항상 생성되고 `axis_status=deferred|not_requested`를 표현하는지 확인
- L10 `catalog_sync_contract_package.m6_context_status`, `sql_context_pack.m6_context_status`, L9 `gate_summary.m6_context_status`가 일치하는지 확인
- L6 Silver/Gold spec이 `write_mode=preview_only`인지 확인
- `preview_scope`에 legacy `window_id`가 없는지 확인
- 모든 생성 JSON의 artifact id 기반 `*_ref`가 `artifact_reference_manifest.json`에서 resolve되는지 확인
- approved Gold에서 aggregate params가 v2.1.1 구조를 따르는지 확인
- main 호환 export 4종이 생성되는지 확인

실패 3건은 이번 M3 파일 변경이 아니라 기존 Windows 실행환경과 Week2/M5 쪽 테스트 기대값에서 발생했다.

- `test_week2_workflow_run_can_execute_spark_runner_and_catalog_parquet_output`: local Spark runner 실행환경에서 run status가 `failed`
- `test_week2_catalog_metadata_tracks_latest_successful_run`: Windows path separator 때문에 `endswith("reviews/gold/...")` 기대값과 불일치
- `test_week2_run_and_catalog_survive_service_restart`: 동일하게 Windows path separator 기대값 불일치

이번 PR은 M3만 올리는 목적이므로 이 실패를 고치기 위해 M5/M2 코드를 변경하지 않았다.

## 6. 남은 한계

- `m3_l0_l6_spark_contract_pipeline.py` 파일명은 과거 호환성 때문에 유지했다. 새 canonical 실행 파일은 `tools/m3_l0_l10_spark_contract_pipeline.py`다.
- Spark/MinIO 대용량 검증 하네스는 M3 계약 검증용이며, 실제 제품 책임은 M2/M5와 연결해야 한다.
- Codex CLI 모델 호출은 L4의 `ai_model_slot`과 외부 audit hook으로 분리했다. 단위 테스트에서는 외부 모델 호출을 하지 않고 deterministic fallback 산출물의 계약 정합성을 검증한다.
